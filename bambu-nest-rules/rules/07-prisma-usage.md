# 07 — PrismaService from @app/prisma

Database access goes through `PrismaService` injected from the `@app/prisma` dynamic module. Never import `PrismaClient` directly from `generated/prisma`, and never instantiate a client.

## Module registration

Each app registers Prisma once in its root module:

```ts
// apps/<service>/src/<service>.module.ts
import { PrismaModule } from '@app/prisma';
import { DATABASE_URL } from '@config/env.config';

@Module({
  imports: [
    PrismaModule.forRoot({ connectionString: DATABASE_URL }),
    // ...
  ],
})
export class ExampleModule {}
```

`PrismaModule` is registered as `global: true`, so nested feature modules do **not** re-import it. Add it only in the app's root module.

## Service usage

```ts
import { PrismaService } from '@app/prisma';
import { Prisma } from 'generated/prisma/client';

@Injectable()
export class PostsService {
  constructor(private readonly prisma: PrismaService) {}

  async create(dto: CreatePostDto) {
    return this.prisma.post.create({
      data: { /* ... */ },
      select: this.buildSelect(),
    });
  }
}
```

## Rules

1. **Inject `PrismaService`** — do not `new PrismaClient()`, do not pull the raw client from `generated/prisma/client`.
2. **Register `PrismaModule` once per app** in the root module. Feature modules reuse the global instance.
3. **Prefer `select` over `include`** when returning data to callers. It avoids over-fetching and bakes in the exact response shape. Use the `satisfies Prisma.<Model>Select` pattern like `apps/posts/src/posts.service.ts`.
4. **Types come from `generated/prisma/client`** — import `Prisma`, model types, and enums (`UserRole`, `StreamStatus`, `PostVisibility`, `FileType`) from there.
5. **Never edit `generated/prisma/*`** — it is regenerated by `npm run prisma:generate` after every `schema.prisma` change.
6. **Multi-step writes use `this.prisma.$transaction`** — either the array form for independent writes or the callback form when reads depend on prior writes.
7. **Pagination uses the DTO helpers** — `const { skip, take } = query.buildPagination();` then `query.buildMeta(total)` for the response metadata (see rule 04).
8. **Fan-in reads: use `Promise.all([findMany, count])`** so pagination lists do not do sequential round-trips.

## Transactions

```ts
await this.prisma.$transaction(async (tx) => {
  const post = await tx.post.create({ data });
  await tx.postMedia.createMany({ data: media });
  return post;
});
```

Keep transactions short. Never `await` an external HTTP call from inside the callback.

## Building `select` trees

Centralize the select in a private helper on the service when you reuse it in multiple methods:

```ts
private buildSelect() {
  return {
    id: true,
    body: true,
    artist: { select: { id: true, slug: true } },
    _count: { select: { likes: true } },
  } satisfies Prisma.PostSelect;
}
```

`satisfies` preserves the exact literal type so callers get precise return types without drifting.

## Anti-patterns

```ts
// ❌ Direct client
import { PrismaClient } from 'generated/prisma/client';
const prisma = new PrismaClient();

// ❌ Re-importing PrismaModule inside a feature sub-module
@Module({ imports: [PrismaModule.forRoot({ connectionString: DATABASE_URL })] })
export class OrdersSubModule {} // ❌ already global

// ❌ Over-selecting via include when you only need two fields
this.prisma.post.findMany({ include: { artist: true, media: true } });
```
